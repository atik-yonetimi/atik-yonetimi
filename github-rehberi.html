Bu sayfa, GitHub bilmeyen ekip üyelerinin projede sorunsuz çalışması için hazırlandı. Amaç: herkes aynı düzeni uygulasın, lider olarak ben de süreci rahat yönetecek olayım.

0) Tek seferlik örnek senaryo (kronolojik)

Bu senaryoyu bir kez anlattığımızda herkes “Issue → Branch → PR → Onay” mantığını oturtacak.

ŞerifIssue #12: “Flutter Login”
↓
Branchgit checkout -b feature/flutter-login
Yeni çalışma alanı açar (main bozulmaz)
↓
Commit + Pushgit add . → değişikliği seçer
git commit -m "add: ..." → kaydeder
git push → GitHub’a gönderir
↓
PR açGitHub’da “Pull Request” aç
Liderden inceleme (review) iste
↓
MehmetReview → onay → merge (main güncellenir)

Bu şemadaki komutlar (terminal kullananlar için)

GitHub Desktop kullanıyorsanız aynı işlemler butonlarla yapılır; isimleri aynıdır (Commit / Push / Pull Request).

Komut Kısaca ne yapar?
git clone <repo_url> Projeyi GitHub’dan kendi bilgisayarına indirir (ilk kez).
git checkout -b feature/... Yeni branch açar ve ona geçer (işini burada yaparsın).
git add . Değişiklikleri commit’e dahil etmek için seçer.
git commit -m "..." Seçili değişiklikleri “kayıt” eder (geri dönülebilir adım).
git push Branch’ini GitHub’a gönderir (PR açabilmen için şart).
Sıra Kim? Ne yapıyor? Çıktı
1 Mehmet (Owner) GitHub’da Issue açar: #12 Flutter Login ve kabul kriterini yazar. İş tanımı netleşir
2 Şerif Repoyu kendi bilgisayarına çeker (clone) ve main’den yeni branch açar: feature/flutter-login. Güvenli çalışma alanı (branch)
3 Şerif Kodu değiştirir, küçük commit’ler atar. Örn commit mesajı: add: driver login screen. Değişiklikler kayıt altına alınır
4 Şerif Branch’ini GitHub’a gönderir (push) ve PR açar. PR açıklamasına 2 satır yazar: “Ne değişti?” + “Nasıl test ettim?” Pull Request oluşur
5 Mehmet PR’ı inceler: kod mantığı + isimlendirme + kabul kriteri. Gerekirse “şunu düzelt” diye yorum bırakır. Review geri bildirimi
6 Şerif İstenen düzeltmeleri yapar, tekrar push eder. (PR otomatik güncellenir.) PR güncellenir
7 Mehmet Her şey uygunsa PR’ı merge eder. Issue’yu kapatır. Değişiklikler main’e girer
8 Ekip (Herkes) Kendi bilgisayarında main’i günceller (pull/sync). Sonra yeni işe branch açar. Herkes güncel kodla devam eder
Lider notu: Bu senaryodaki en kritik iki nokta:
  • Main’e direkt commit yok. Her iş branch + PR.
  • PR açıklaması boş olmayacak. “Ne yaptın?” ve “Nasıl test ettin?” yazılacak.

1) GitHub’ı 3 cümlede anlayalım

  • Repository (Repo): Projenin tüm kodunun durduğu yer.
  • Branch: Kodun ayrı bir kopyası gibi düşünebilirsin. İşini burada yaparsın, ana kodu bozmazsın.
  • Pull Request (PR): “Benim yaptığım değişiklikleri ana koda ekleyelim mi?” diye inceleme isteği.
Lider notu: Bizim ekipte “ana kod” = main. Main’e kimse direkt commit atmayacak. Her şey branch + PR.

Bu akış dışına çıkmayın. Çıkarsanız aynı hatayı 2–3 gün geri sararak düzeltmek zorunda kalıyoruz.

Lider notu: PR açmadan “bitti” yok. PR = yaptığın işin hem kanıtı hem de güvenli kontrol noktası.

3) Hangi aracı kullanacağız?

A) En kolay yol: GitHub Desktop (önerilen)

  • Görsel arayüz, hata yapma ihtimali az.
  • Branch/commit/push/PR açma adımları çok net.

B) Terminal (opsiyonel)

  • Hızlıdır ama yanlış komutla karışabilir.
  • Bu dokümanda “en gerekli” komutlar var.

5) “Ben bir işi nasıl yapacağım?” (adım adım)

Adım 1 — Issue’yu oku

  • Ne istendiğini anla: çıktı ne olacak?
  • Kabul kriteri var mı?

Adım 2 — Branch aç

  • Branch’i main üzerinden oluştur.

Adım 3 — Değişikliği yap

  • Küçük parçalara böl. Tek seferde 50 dosya değiştirme.

Adım 4 — Commit at

  • Commit mesajı kısa ve net: add/fix/update ile başla.

Adım 5 — Push

  • Branch’ini GitHub’a gönder.

Adım 6 — PR aç

  • Başlık: ne yaptın?
  • Açıklama: neden yaptın?
  • Test: nasıl denedin?
Lider notu: PR açıklamasına 2 satır yeter: “Ne değişti?” + “Nasıl test ettim?”

6) PR şablonu (kopyala-yapıştır)

## Özet
- ...

## Ne değişti?
- ...

## Nasıl test ettim?
- [ ] Çalıştırdım / build aldım
- [ ] Temel akışı denedim

## Not
- (Varsa) Risk / eksik / takip işi
Lider notu: “Test ettim” yazıp test etmemek en pahalı hata. En azından temel akışı bir kez çalıştırın.

7) Çakışma (Conflict) nedir?

Aynı satırı iki kişi değiştirirse GitHub “hangisi doğru?” diye sorar. Bu normal.

Senin Branchsen değişiklik yaptın
↓
mainbaşkası aynı yerleri değiştirdi
↓
ConflictGitHub “hangisi doğru?” diye sorar
↓
Çözümsatırları seç + test et + PR devam
Ne yap? Ne yapma?
  • Önce lideri haberdar et.
  • Değişiklikleri satır satır oku.
  • Çözünce uygulamayı/servisi bir kez çalıştır.
  • Rastgele “Accept all” yapma.
  • Test etmeden merge isteme.
Lider notu: Conflict görürsen paniğe gerek yok. “Ne değişti?” sorusunu birlikte çözeriz.

9) En önemli 10 kural (kısa checklist)

Yap Yapma
  • ✓ Her iş için branch aç
  • ✓ Küçük commit’ler at
  • ✓ PR aç ve açıklama yaz
  • ✓ PR’da nasıl test ettiğini yaz
  • ✓ Takıldığında erken sor
  • ✗ main’e direkt commit atma
  • ✗ PR’sız “bitti” deme
  • ✗ Conflict’te rastgele seçim yapma
  • ✗ Büyük değişikliği tek PR’da yığma
  • ✗ Test etmeden merge isteme
