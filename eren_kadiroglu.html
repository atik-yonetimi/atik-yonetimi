PROJE ADI:
Akıllı Şehir Çoklu Atık Türü Yönetimi ve Rota Optimizasyonu

SİSTEM ÖZETİ (data layer açısından):
Backend “aktif rota” ve “duraklar” listesini döner. Mobil kullanıcı (sürücü) duraklarda done/skipped ile kayıt açar. Saha koşullarında internet kesilebilir; bu yüzden offline-first gerekir: rota cache’lenir, collection kayıtları kuyruklanır, internet gelince senkronlanır.

KESİN KURALLAR:
1) Türler: Cam, Plastik, Kağıt, İkinci El Eşya, Metal.
2) Sürücü sadece kendi aracının aktif rotasını görür.
3) Durak durumları: pending / arrived / done / skipped.
4) Done = collection kaydı var (miktar kg zorunlu).
5) Skipped = sebep zorunlu.
6) Offline senaryosu “normal” kabul edilir (özellikle demo/pilot).

BENİM ROLÜM:
Eren Kadiroğlu — Flutter Mobil Geliştirici + Tester.

BENİM SORUMLULUKLARIM:
A) API client + error handling:
   - auth/login, me, routes/active, stops/status, collections
   - timeout, 401/403, 500 gibi hatalarda kullanıcıya net mesaj
B) Local cache:
   - aktif rota + stop listesi lokal DB’de saklanacak
   - app kapanıp açılınca rota kaybolmayacak
C) Offline queue:
   - collection kayıtlarını “pending” olarak sakla
   - online olunca sırayla gönder
   - başarılı olunca “sent”; hata olunca “failed” + retry
   - duplicate önleme (idempotency) düşün
D) Test:
   - demo öncesi uçtan uca checklist
   - bug raporu tekrar üretilebilir formatta

VERİ MODELİ (mobil local DB için minimum öneri):
- cached_route: route_id, vehicle_plate, waste_type, version(optional), fetched_at
- cached_stops: stop_id, route_id, sequence_no, container_id, lat, lng, status
- outbox_collections: local_id, stop_id, result(done/skipped), amount_kg, skip_reason, collected_at, gps_lat, gps_lng, status(pending/sent/failed), attempts, last_error
- idempotency_key: (örn. local_id veya stop_id+timestamp hash)

AI’DEN İSTEDİĞİM (NET ÇIKTI):
1) Flutter’da offline-first yaklaşım:
   - hangi katmanlar? (repository, data source, cache, outbox)
2) Queue state machine:
   - pending → sending → sent
   - pending/failed → retry
   - maksimum deneme + backoff öner
3) Duplicate önleme:
   - client-side idempotency_key nasıl üretilir?
   - aynı stop için 2 kayıt yanlışlıkla gitmesin
4) Senkron stratejisi:
   - network geri gelince tetikleme
   - uygulama açılışında “outbox flush”
5) Test planı:
   - 10 maddelik demo checklist (login → rota → done/skipped → offline → senkron)
   - 5 maddelik “en sık hata” listesi + nasıl yakalanır
6) Kullanıcı metinleri:
   - offline uyarısı, gönderim başarılı/başarısız, tekrar denenecek mesajları (kısa Türkçe)
